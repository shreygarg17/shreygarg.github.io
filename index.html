// The Problem

Approach

The system was decomposed into three focused services with clear responsibilities. The OTP Service owned generation (cryptographically secure, short-lived codes), secure storage (Redis with TTL), delivery routing, and verification + token issuance. The Rate Limiter ran as a middleware layer — configurable per endpoint — using sliding window counters stored in Redis to enforce per-user and per-IP limits. The Communication Layer abstracted over multiple providers with automatic failover, ensuring delivery SLAs.

Design Decisions & Trade-offs
CHOSE
Redis for OTP storage over DB — sub-millisecond reads at scale with automatic TTL-based expiry; eliminated manual cleanup jobs
TRADE
Redis data loss risk on restart → mitigated with persistence config + graceful fallback to DB for verification
CHOSE
Sliding window rate limiting over token bucket — smoother request distribution, fairer per-user enforcement across time boundaries
CHOSE
Configurable limits per endpoint — allowed tuning without deploys; OTP endpoints tighter (3/min) vs login (10/min)
CHOSE
Multi-provider comm abstraction — automatic fallback between SMS providers ensured delivery without code changes
Impact

The system successfully handled Zepto's authentication traffic at scale, surviving peak load events without degradation. Abuse attempts (OTP flooding, bot signup) were blocked at the rate limiting layer before reaching downstream services, reducing wasted SMS costs and protecting system stability. Onboarding reliability improved measurably, directly contributing to conversion rates for new user flows.
